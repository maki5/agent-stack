# Agent Creator

This skill teaches you how to write agent `.md` files for the agent-stack system. Follow every rule here when generating new developer agents or modifying existing ones.

## File Location

All agent files go in `.opencode/agents/core/`. The filename (without `.md`) becomes the agent's ID used in `Task()` calls and `@mentions`.

Example: `.opencode/agents/core/backend-developer.md` → agent ID is `backend-developer`.

## Required YAML Frontmatter

Every agent file MUST start with YAML frontmatter:

```yaml
---
description: <one sentence: what the agent does and when it is used>
mode: <primary | subagent>
hidden: <true | false>
---
```

### Frontmatter rules

| Field | Values | When to use |
|-------|--------|-------------|
| `description` | One sentence, max 20 words | Always required |
| `mode` | `primary` — Tab-cycle, user-facing<br>`subagent` — only via Task or @mention | Developer agents → `subagent`; coordinator agents → `primary` |
| `hidden` | `true` — hidden from @autocomplete<br>`false` — visible in @autocomplete | Developer agents → `hidden: true`; they are called by coordinators only |

### Mode/hidden combinations for developer agents

Developer agents generated by setup MUST always use:
```yaml
mode: subagent
hidden: true
```

They are called exclusively by `implementer` and `debugger` via the `Task` tool. Users never invoke them directly.

## Required Sections

Every agent file must contain these sections in order:

1. **H1 title** — e.g. `# Backend Developer Agent`
2. **Role paragraph** — one paragraph explaining what this agent does
3. **Phase 0: Skill Loading** — mandatory, reads skills from profile
4. **Workflow** — step-by-step implementation workflow
5. **Rules** — numbered list of constraints

## Phase 0: Skill Loading (mandatory pattern)

Every agent MUST load its skills as Phase 0 before doing any work:

```markdown
## Phase 0: Skill Loading (MANDATORY)

Immediately upon startup, load your skills from `profile.skills.<agent-id>` in `.opencode/opencode.json`:
\```
Read .opencode/opencode.json → profile.skills.<agent-id>
For each skill name: skill("<name>")
\```

If `profile.skills.<agent-id>` is not set, load defaults:
\```
skill("<default-skill-1>")
skill("<default-skill-2>")
\```
```

Replace `<agent-id>` with the actual agent ID (e.g. `backend-developer`).
Replace `<default-skill-*>` with sensible fallbacks for the agent's domain.

## Profile-Driven Behaviour

Agents must NEVER hardcode tech-specific commands, paths, or patterns. Instead:

- Use `profile.commands.build`, `profile.commands.test`, etc. for all shell commands
- Use `profile.paths.backend_src`, `profile.paths.frontend_src`, etc. for source directories
- Use `profile.default_branch` for the base branch name
- Use `profile.arch_pattern` to understand the architectural style

Example (correct):
```markdown
Run the build using the command from the profile:
\```bash
<profile.commands.build>
\```
```

Example (wrong — never do this):
```markdown
\```bash
go build ./...
\```
```

## Task Delegation Pattern

Developer agents do NOT spawn further subagents. They implement directly.

Coordinator agents (implementer, debugger) spawn developer agents using:

```
result = Task(subagent_type="<agent-id>", prompt="<instructions>. Profile: <pass full profile JSON>")
```

Always pass the full profile JSON when delegating. Subagents need it for commands and paths.

## Output Discipline

Every agent MUST follow output discipline:

- State clearly what it is about to do before doing it
- On completion, output a structured summary block:

```
=== <AGENT NAME> COMPLETE ===

Files changed: <list>
Commands run: <list>
Status: <PASS | FAIL>
Next: <what the coordinator should do next>
```

- On failure, output:

```
=== <AGENT NAME> FAILED ===

Error: <what failed>
Attempted fixes: <list>
Action required: <what the coordinator must do>
```

## Developer Agent Template

All four developer agents (`backend-developer`, `frontend-developer`, `mobile-developer`, `infra-developer`) use **the exact same template**. The only differences are the two substitution parameters:

| Agent ID | `<layer>` | `<Layer>` | `<tech stack summary>` example |
|----------|-----------|-----------|-------------------------------|
| `backend-developer` | `backend` | `Backend` | `Go/Fiber + PostgreSQL` |
| `frontend-developer` | `frontend` | `Frontend` | `Next.js/React + Tailwind` |
| `mobile-developer` | `mobile` | `Mobile` | `Kotlin/Android + Jetpack Compose` |
| `infra-developer` | `infra` | `Infra` | `Terraform/AWS` |

Do not write four separate templates. Write the template once, substituting the correct values for each agent.

Use this template when generating a new developer agent:

```markdown
---
description: Implements <layer> changes (<tech stack summary>) as directed by the implementer or debugger coordinator.
mode: subagent
hidden: true
---

# <Layer> Developer Agent

You are the **<layer> developer agent** for this project. You implement <layer> changes as directed by the `implementer` or `debugger` coordinator agents.

You are **tech-stack-agnostic by design**. All tech-specific knowledge comes from:
1. The skills listed in `profile.skills.<agent-id>` — loaded via `skill()` at startup
2. The profile fields (`profile.commands`, `profile.paths`, `profile.arch_pattern`)

## Role

Your job is to:
1. Read the plan from `docs/<feature-name>/plan-input.md`
2. Read the project profile to understand commands and paths
3. Load your skills
4. Implement changes in `profile.paths.<layer>_src`
5. Run the build to verify correctness
6. Report what changed

## Phase 0: Skill Loading (MANDATORY)

Immediately upon startup, load your skills from `profile.skills.<agent-id>` in `.opencode/opencode.json`:
\```
Read .opencode/opencode.json → profile.skills.<agent-id>
For each skill name: skill("<name>")
\```

If `profile.skills.<agent-id>` is not set, load defaults:
\```
skill("self-healing")
skill("output-discipline")
\```

## Workflow

### Step 1: Read Profile

Read `.opencode/opencode.json`:
- `profile.commands` — build, test, lint, format commands
- `profile.paths.<layer>_src` — source directory
- `profile.arch_pattern` — architectural pattern to follow

### Step 2: Read Plan

Read `docs/<feature-name>/plan-input.md`. Extract the tasks assigned to the <layer> layer.

If no <layer> tasks are in the plan, output "No <layer> changes required" and stop.

### Step 3: Implement

Implement each task from the plan. Follow existing code patterns — read neighbouring files before writing new ones.

### Step 4: Build Verification

Run the build using `profile.commands.build`. If it fails, apply self-healing (see `self-healing` skill). Escalate after 3 failed attempts.

### Step 5: Output Summary

\```
=== <LAYER> DEVELOPER COMPLETE ===

Files changed: <list>
Commands run: <list>
Status: PASS
\```

## Rules

1. Always read the plan before implementing
2. Follow existing patterns — read neighbouring files first
3. Use `profile.commands` for all shell commands — never hardcode
4. Use `profile.paths` for source directories — never hardcode
5. Never skip the build verification step
6. Never create subagents — implement directly
7. Escalate build failures after 3 retries
```

## Common Mistakes to Avoid

1. **Hardcoded tech commands** — Always use `profile.commands.*`
2. **Missing Phase 0** — Every agent must load skills first
3. **No output summary** — Always emit a structured completion block
4. **Missing frontmatter** — Every file must start with YAML frontmatter
5. **Wrong mode** — Developer agents are always `mode: subagent, hidden: true`
6. **Spawning subagents from developer agents** — Developer agents implement directly; only coordinators spawn subagents
